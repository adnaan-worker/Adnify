# Adnify 单兵 Agent IDE 产品设计

**日期：** 2026-03-10

## 1. 产品定位

Adnify 不再朝“兼容 VS Code 的 AI 编辑器”方向演进，而是明确定位为：

> **面向个人开发者的原生 Agent IDE**

核心价值不是“陪聊式辅助编码”，而是：

> **你给目标，IDE 负责拆解、分工、执行、验证、回滚、总结。**

这意味着 Adnify 的核心体验应从“消息流”切换到“任务流”，并以任务编排、执行治理和安全收敛为中心设计产品。

## 2. 产品原则

### 2.1 默认半自主，可切强自主
- 默认模式是半自主：每个阶段结束后主动汇报一次。
- 用户可切换为强自主模式：仅在高风险、越权、冲突或熔断时打断。

### 2.2 安全机制默认开启，但允许关闭
- 安全机制不依赖零散确认框，而依赖前置的图形化信任策略。
- 系统默认开启安全执行、副本隔离、预算与熔断。
- 用户始终保有关闭或放宽这些限制的权力。

### 2.3 多 Agent 是显式协作，不是隐形子代理
- 多 Agent 应该是可见、可配置、可追踪的专家团队。
- 用户可以手动指定不同专长的 Agent，并使用模板快速生成协作配置。
- 系统不应让多个模型在后台自由改同一份代码。

### 2.4 结构化移交优先于自由对话
- Agent 之间不以群聊式自由交流为主。
- 任务推进依赖工作包、移交包与协调器这三类结构化对象。

### 2.5 熔断是核心能力，不是附属优化
- 系统必须知道“什么时候该停”。
- 一旦出现循环、互相打架、无净进展或预算激增，应主动中断并转入裁决流程。

## 3. 核心对象模型

### 3.1 任务（Task）
一个用户目标对应一个任务实例。任务包含：
- 目标与成功标准
- 计划与阶段
- 风险等级
- 预算（Token、时间、验证成本、外部调用）
- 执行模式（半自主 / 强自主）
- 执行环境（当前工作区 / 隔离任务副本）
- 当前状态（计划中、执行中、验证中、待裁决、已完成、已熔断）

### 3.2 专家 Agent（Specialist Agent）
专家 Agent 是任务中的显式角色，不是隐藏线程。建议 MVP 支持：
- `Frontend Agent`：UI、交互、组件结构、视觉表现
- `Logic Agent`：状态流、边界条件、数据转换、业务逻辑
- `Verifier Agent`：测试、构建、运行、回归、日志确认
- `Reviewer Agent`：Diff 审查、风险识别、建议输出

每个 Agent 都应具备以下配置：
- 模型选择
- 工具权限
- 可写范围
- 风格偏好
- 预算上限
- 默认验证职责

### 3.3 工作包（Work Package）
工作包是多 Agent 协作的最小推进单位。每个工作包至少包含：
- 工作目标
- 指派 Agent
- 目标域（UI / Logic / API / Test / Review）
- 可写范围
- 依赖项
- 预期产物
- 验证要求
- 状态（待分配、待执行、执行中、阻塞中、待合并、已验证）

### 3.4 移交包（Handoff Package）
移交包是 Agent 完成工作后的结构化输出，用于安全接棒与收敛。移交包必须包含：
- 完成内容摘要
- 修改文件列表
- 修改理由
- 关联依赖
- 尚未验证或待处理事项
- 建议下一个接手者
- 附属产物（Diff、日志、截图、测试结果）

### 3.5 协调器（Coordinator）
协调器不是普通 Agent，也不是用户手动复制代码。它是系统级、受限、可审计的收敛角色，负责：
- 根据任务与模板拆分工作包
- 检查依赖与写范围
- 在同步点收敛多个移交包
- 发现冲突并发起裁决
- 触发验证
- 将结果安全回灌到任务集成线

## 4. 信任与安全模型

### 4.1 三层生效范围
信任策略按三层生效：
- 全局默认
- 工作区覆盖
- 任务临时覆盖

### 4.2 图形化信任模式
MVP 提供预设模式，避免用户配置底层规则：
- 安全模式
- 平衡模式
- 自主模式
- 完全手动模式

### 4.3 可配置维度
图形化信任设置应覆盖：
- 文件改动范围
- 命令执行权限
- Git 写权限
- 网络与外部副作用权限
- 执行环境偏好
- 中断策略

### 4.4 默认开启的安全机制
默认情况下系统应启用：
- 写范围保护
- 执行预算
- 熔断检测
- 隔离任务副本
- 协调器收敛门禁

## 5. 执行环境模型

### 5.1 当前工作区
适用于小改动、低风险、快速交互。

### 5.2 隔离任务副本
适用于中大任务、跨文件修改、依赖安装、结构重构、长链路验证。

“隔离任务副本”是产品概念，不要求用户理解底层实现。底层可根据项目类型选择：
- Git 项目：优先 worktree
- 非 Git 项目：优先临时目录副本

### 5.3 默认策略
- 小改动：当前工作区
- 中大改动：自动切换隔离任务副本
- 用户可以全局/工作区/任务级别覆盖该规则

## 6. 多 Agent 并行安全模型

### 6.1 基本原则
多 Agent 的并行不是“共同编辑同一份代码”，而是：

> **在依赖图约束下并行推进多个工作包。**

### 6.2 并行判定条件
系统判断是否可并行，必须同时看：
- 写范围是否重叠
- 目标域是否冲突
- 是否存在依赖关系

### 6.3 默认规则
- 允许并行：不同模块、不同职责面、不同写范围
- 谨慎并行：同一功能链路但不同层（如 UI 与状态层）
- 默认串行：同一状态源、同一 API 契约、同一核心文件、结构性重构

### 6.4 同步点
并行任务不应无限并行到底。系统必须定义同步点，在同步点统一进行：
- 依赖检查
- 冲突扫描
- 中间验证
- 是否允许进入下一阶段

## 7. 协作与移交流程

### 7.1 Agent 之间的交互类型
不建议自由对话式互聊，建议三种结构化交互：
- 请求依赖
- 提交产物
- 发起阻塞

### 7.2 回灌规则
移交结果不能直接写回主任务线，必须先经过协调器：
- 写范围授权检查
- 冲突检查
- 依赖满足检查
- 验证门禁检查

### 7.3 用户的角色
用户不再是代码搬运工，而是在关键节点充当裁决者：
- 接受/拒绝移交包
- 只接受部分 patch
- 退回重做
- 改派其他 Agent 复审
- 切换信任模式或执行模式

## 8. 熔断与成本控制系统

### 8.1 为什么必须有熔断
多 Agent 系统最容易失控的不是单次错误，而是：
- 单 Agent 循环
- Agent 互相打回
- 重复验证
- 重复命令执行
- 无净进展但持续消耗预算

### 8.2 五层熔断
建议 MVP 支持五层熔断：
1. 工作包连续失败熔断
2. 单 Agent 循环熔断
3. Agent 互相打架熔断
4. 任务预算熔断
5. 全任务无净进展熔断

### 8.3 熔断后的系统行为
熔断不是任务失败，而是状态转移：
- 待裁决
- 待缩小范围
- 待更换策略

### 8.4 熔断报告
熔断触发后系统必须生成结构化报告，展示：
- 触发对象
- 触发原因
- 最近关键动作
- 已消耗成本
- 当前阻塞点
- 建议下一步

## 9. UI 设计方向

### 9.1 主视图：任务编排看板
主入口不应是多聊天窗，而应是任务编排看板。

### 9.2 关键视图
MVP 至少需要以下视图：
- 任务总览区
- 专家 Agent 阵容条
- 工作包看板
- 移交包面板
- 协调器集成视图
- 信任/执行环境控制条
- 熔断与预算面板

### 9.3 交互原则
- 任务状态优先于消息历史
- 结构化产物优先于自然语言解释
- 风险和冲突必须可视化
- 用户只在关键节点介入

## 10. MVP 范围

### 10.1 必须先做的能力
第一阶段建议只做以下能力：
1. 任务对象与任务看板
2. 图形化信任中心
3. 隔离任务副本机制
4. 工作包 / 移交包模型
5. 协调器收敛与冲突检查
6. 熔断与预算控制

### 10.2 第二阶段再做
- 浏览器自动化验证
- 更丰富的专家模板
- 更强的验证 Agent
- 更细的模型路由与成本优化

## 11. 非目标
以下方向不是当前产品阶段的重点：
- VS Code 扩展兼容
- Marketplace 生态复刻
- 团队协作优先的审查/审批流
- 远程开发全家桶
- 单纯 UI 美化或安装包瘦身

## 12. 结论
Adnify 的正确方向不是“更像现有编辑器”，而是成为：

> **一个默认安全、可配置自主度、支持多专家并行、知道何时停下来的个人 Agent IDE。**

真正的差异化不在于它能调用多少模型，而在于：
- 能否把长任务变成结构化流程
- 能否让多 Agent 协作不互相踩踏
- 能否在成本和风险失控前主动熔断
- 能否让用户始终保持最终裁决权
